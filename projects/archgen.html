Overview

archgen takes a one-sentence software idea and runs it through a sequence of specialist AI agents — each focused on a single concern — and produces a full architecture blueprint covering system design, storage, security, deployment, performance, and cost.

A final critic agent reviews the combined output and flags gaps, overengineering, and risks the other agents missed.

Built as a way to learn multi-agent orchestration, prompt design across domains, and working with several LLM provider APIs in one project.

Pipeline

The flow for every request is the same regardless of domain: 

idea → extract requirements → classify domain → run specialist agents → critic review → export

Output streams live to the frontend via Server-Sent Events and can be exported as Markdown when the run is complete.

Agents

Each agent is a separate file responsible for one concern. Prompts and provider calls are explicit — no LangChain abstraction.

  • Extractor — pulls structured requirements from the raw idea
  • Architect — produces the system design
  • Database — storage strategy and data model
  • Security — threat model and security controls
  • DevOps — deployment and infrastructure plan
  • Performance — bottleneck analysis and scaling notes
  • Cost — infrastructure cost estimate
  • Critic — reviews all output, flags what was missed
  • Domain experts — nine additional agents for non-web domains

Domains

Each domain is a self-contained plugin that injects domain-specific context into every agent. Adding a new domain is one new plugin file.

  • Web application
  • Mobile application
  • Game engine
  • OS kernel
  • Compiler
  • Distributed system
  • Database engine
  • AI / ML platform
  • Embedded firmware
  • HPC / research software

Stack

  • Backend — Python, FastAPI, Server-Sent Events for streaming
  • Frontend — Vanilla JS, HTML, CSS — no build step
  • LLM providers — Gemini, Groq, Cerebras, OpenRouter, Claude
  • Deployment — Dockerfile included

Each agent is independently assigned to whichever provider suits it best via a single config file. The pipeline is designed to run entirely on free-tier API keys — no credit card required to try it.

Structure

  • agents/ — one file per specialist agent
  • plugins/ — one file per domain
  • providers/ — thin wrappers for each LLM provider
  • config/ — agent-to-model assignments
  • core/ — pipeline orchestration and domain classifier
  • api/ — FastAPI routes and SSE streaming
  • frontend/ — vanilla JS/HTML/CSS interface
  • output/ — Markdown blueprint export

Usage

git clone https://github.com/notsajeed/archgen
cd archgen
pip install -r requirements.txt
cp .env.example .env
# paste at least one API key into .env
uvicorn main:app --reload

Open http://127.0.0.1:8000, type a software idea, and watch the pipeline run. Groq has the highest free daily cap (1,000 req/day) and is the easiest key to start with.

Development Notes

The plugin system was the core design decision — each domain injects its own context so agents don't produce irrelevant suggestions (a compiler doesn't get told to add a REST API).

Runs on free-tier LLM APIs with real daily quotas. Hitting a quota error mid-run is expected behaviour on the free tier, not a bug — swap that agent to a different provider in config/agent_models.py.

No per-user auth or multi-tenancy — designed to be run locally with your own keys.
